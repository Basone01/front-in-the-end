นี่คือข้ออ้างที่ทั้ง track นี้ตั้งอยู่บนมัน: **architecture พื้นฐานของเว็บ — document, link, และ form — คือ application platform ที่สมบูรณ์และใช้งานได้จริง** ส่วนเทคโนโลยีทุกอย่างที่เราจะได้เจอหลังจากนี้ (ไม่ว่าจะ SPA, SSR, hydration, หรือ islands) ล้วนเกิดขึ้นมาเพื่ออุดช่องโหว่เฉพาะบางอย่างที่ document ทำเองไม่ได้ และถ้าคุณยังไม่รู้แน่ชัดว่า document ทำอะไรได้บ้าง คุณก็จะไม่มีทางรู้ว่าจริง ๆ แล้วคุณมีช่องโหว่อะไรอยู่ — แล้วก็จะเผลอไปซื้อตัวอุดมาปะรอยรั่วที่ไม่เคยมีอยู่ตั้งแต่แรก

frontend engineer ส่วนใหญ่เรียนรู้ stack แบบจากบนลงล่าง: ถูกจ้างเข้ามาทำในโค้ดเบส React อยู่กับ framework ก่อน แล้วค่อยมารู้จัก platform ทีหลัง หรือบางทีก็ไม่เคยรู้จักเลย นั่นไม่ใช่ความผิดบาปอะไร มันเป็นแค่ลำดับที่อุตสาหกรรมเลือกจะสอนกันในยุคนี้ แต่มันสร้างจุดบอดเฉพาะอย่างหนึ่งขึ้นมา: คุณคว้า `useState` มาใช้ก่อนจะถามตัวเองว่า state นั้นมีอยู่ที่อื่นแล้วหรือเปล่า และเรียกหา framework ก่อนจะถามว่า browser มีของพวกนี้ให้ใช้อยู่แล้วไหม

บทแรกจึงเป็นการพาทัวร์ platform ที่คุณ ship งานอยู่บนมันมาตลอด ไม่ว่าคุณจะรู้ตัวหรือไม่ก็ตาม

## ระบบจัดการ state แบบดั้งเดิม

"เว็บแอป" ในปี 1995 มี state model ที่สมบูรณ์อยู่แล้ว และมันคุ้มที่จะอธิบายให้ชัด เพราะมันยังเป็นพื้นฐานอยู่ใต้ทุกอย่างที่คุณสร้างทุกวันนี้:

- **URL คือ client state** — คุณอยู่หน้าไหน กำลังดูสินค้าตัวไหน หรืออยู่ที่ผลการค้นหาหน้าที่เท่าไร ทั้งหมดถูกเข้ารหัสไว้ใน URL มันจึง share ได้ bookmark ได้ และกดปุ่ม back ได้ ฟรี ๆ
- **Server คือ source of truth** — ตะกร้าสินค้า บัญชีของคุณ จำนวนของในคลัง ทุกอย่างรวมอยู่ที่เดียว และการโหลดหน้าแต่ละครั้งก็คือการอ่านมันใหม่สด ๆ
- **Form คือการ write** — `<form>` คือการเปลี่ยน state ที่มี UI มาให้พร้อมในตัว: มันรับ input, validate, serialize, แล้วส่งไป จากนั้น server ก็ตอบกลับมาเป็นสถานะใหม่ของระบบในรูปของ document หน้าใหม่

ลองสังเกตสิ่งที่*หายไป*จาก model นี้: client-side state ที่อาจไม่ตรงกับ server ได้ ใน architecture แบบนี้มี source of truth อยู่เพียงชุดเดียวเท่านั้น และ browser ก็แค่ render ภาพ snapshot ของมันออกมา bug กองมหาศาลที่คุณต้องสู้กับมันทุกวัน — cache เก่าค้าง, optimistic update ที่พลาด, component สองตัวที่เถียงกันว่า modal เปิดอยู่หรือเปล่า — **มันเกิดขึ้นไม่ได้เลย**ใน architecture แบบนี้ เราไม่ได้กำจัด bug พวกนั้นทิ้งตอนที่เว็บเติบโตขึ้น ตรงกันข้าม เราเป็นคน*นำมันเข้ามา*เองเพื่อแลกกับอย่างอื่น และเนื้อหาที่เหลือของ track นี้ก็คือเรื่องที่ว่า การแลกแบบนั้นคุ้มตอนไหน

## สิ่งที่ HTML กับ CSS ทำได้โดยไม่ต้องพึ่งคุณ

browser คือ UI runtime ที่ผ่านสนามรบมาโชกโชนที่สุดเท่าที่เคยมีมา และพฤติกรรมแบบ "แอป" จำนวนมหาศาลก็ถูกสร้างมาไว้ในตัวมันอยู่แล้ว นี่คือรายการคร่าว ๆ ของสิ่งที่ใช้ JavaScript **เป็นศูนย์บรรทัด**:

- **navigation** — link พร้อม cache, prefetch (`rel="prefetch"`), history และตัวบอกสถานะกำลังโหลด
- **UI แบบเปิด/ปิด** — `<details>`/`<summary>` คือ accordion ที่กดด้วย keyboard ได้และมี accessibility semantics ในระดับที่ปกติต้องพึ่ง library ภายนอกถึงจะทำได้เทียบเท่า
- **overlay** — attribute `popover` ให้คุณสร้างป๊อปอัปที่ปิดได้พร้อมจัดการ focus ให้เสร็จ ส่วน `<dialog>` ก็ไว้ทำ modal
- **การ validate form** — `required`, `type="email"`, `pattern`, `min`/`max` มาพร้อม UI แจ้ง error, การจัดการ focus และ hook สำหรับ style อย่าง `:user-valid`/`:user-invalid`
- **การ render แบบมีเงื่อนไข** — CSS `:checked`, `:has()` และ sibling selector แสดงหรือซ่อน UI ตาม input ที่ผู้ใช้กรอกได้ (ใช่ นี่แหละคือ state-driven rendering — state ซ่อนอยู่ใน input element ที่ browser เป็นคนจัดการให้)
- **ลูกเล่นอื่น ๆ** — smooth scroll, sticky header, carousel (`scroll-snap`), animation (`@keyframes`, view transitions), theme (`prefers-color-scheme`)…

อย่าเพิ่งเชื่อแค่รายการนี้ ทุกอย่างข้างล่างนี้เป็น HTML กับ CSS ล้วน ๆ — ลองเปิด devtools ขึ้นมาแล้วมองหา script ที่ไม่มีอยู่จริงดู:

## อ่านเรื่องนี้ในมุมของ React developer

ถ้างานประจำวันของคุณคือ React ลองแปล document model ให้เป็นคำศัพท์ของคุณดู เพราะมันแมปกันได้เป๊ะ ๆ:

- การโหลดหน้าหนึ่งครั้งก็คือ **render** ครั้งหนึ่ง: server คือ component ที่รับ request เข้ามาแล้วคืน UI กลับออกไป
- URL คือ **lifted state** — ถูกยกขึ้นไปไกลถึงนอก component tree เข้าไปอยู่ใน chrome ของ browser
- การ submit form คือ **state transition**: input event, reducer ที่ฝั่ง server, แล้วก็ re-render

นี่แหละคือเหตุผลที่ React Server Components และ framework อย่าง Remix ถึงให้ความรู้สึก "ใหม่แต่ก็คุ้น ๆ": จริง ๆ แล้วมันกำลังประกอบ document model ขึ้นมาใหม่ในคำศัพท์ของ React ต่างหาก `<Form>` ใน Remix ก็คือ `<form>` ที่ถอดแฟลชตอนรีโหลดหน้าออกไป และ server component ก็คือหน้า PHP ที่เรียนรู้การ compose เป็น สรุปคือ model ของ platform ไม่เคยผิด — มันแค่*ยังไม่ครบ* และเราจะใช้เวลาในบทถัด ๆ ไปตามหาให้เจอว่ามันขาดตรงไหนกันแน่

## แล้วเมื่อไรที่ document ก็เพียงพอแล้วจริง ๆ

ขอพูดความเห็นตรง ๆ เลย: **ถ้าโปรดักต์ของคุณคือเนื้อหาที่คนเข้ามาอ่าน หรือ transaction สั้น ๆ ที่คนเข้ามาทำให้จบ document architecture ไม่ใช่ทางเลือกสำรอง — แต่มันคือทางเลือกที่ถูกต้องอยู่แล้ว** และการเอา client-side application ไปแปะทับมันก็มีแต่จะทำให้ช้าลง เปราะลง และสร้างยากขึ้น

ให้ชัดขึ้นอีก document ชนะเมื่อ:

- **session สั้น** — ไม่มีใคร amortize ต้นทุน bundle 400 KB ลงบนการเข้าชมแค่ 40 วินาทีได้คุ้มหรอก
- **ความจริงอยู่ที่ server อยู่แล้ว** — ถ้าทุก action ที่มีความหมายต้องวิ่งไปหา server (checkout, ค้นหา, เผยแพร่) เจ้า client state ก็เป็นแค่ cache อีกชั้นที่คุณต้องมาคอยทำให้มันซื่อสัตย์
- **คนแปลกหน้าสำคัญ** — search engine, ตัว preview ลิงก์, คนที่เข้ามาครั้งแรกบน wifi โรงแรม คนพวกนี้บริโภค HTML ไม่ใช่แอปที่ hydrate เสร็จแล้วของคุณ

บล็อก, เว็บ documentation, เว็บข่าว, form ราชการ, หน้า browse ของร้านค้า: เป็น document ได้ทั้งหมด และไม่ใช่ในความหมายว่า "เป็น document ไปก่อนจนกว่าจะมีปัญญาสร้างใหม่ให้ถูกต้อง" — แต่เป็น document *ในฐานะที่มันคือการสร้างที่ถูกต้องอยู่แล้ว*

## กำแพงอยู่ตรงไหน

ถึงอย่างนั้นก็เถอะ ลองเอา document ล้วน ๆ ไปสร้างอะไรพวกนี้ดู แล้วคุณจะชนกำแพงเข้าอย่างจัง:

- อะไรก็ตามที่ update **เองโดยที่ผู้ใช้ไม่ได้ร้องขอ** — ราคาแบบสด ๆ, presence, notification
- อะไรที่มี**ลูป feedback เร็ว ๆ** — typeahead search, drag-and-drop, canvas สำหรับวาดรูป
- อะไรที่ผู้ใช้**ค่อย ๆ สะสม state ขึ้นมาจาก interaction เล็ก ๆ หลายครั้ง** — panel filter, editor หลายขั้นตอน, หน้าจอเลือกที่นั่ง — ซึ่งถ้าต้องรีโหลดทั้งหน้าทุกครั้งต่อหนึ่ง interaction มันก็จะทำลายประสบการณ์จนหมด

จุดอ่อนของ document model ก็คือจุดแข็งของมันที่ถูกพลิกกลับด้านพอดี: หน่วยเดียวที่มันใช้ในการ update คือ*ทั้งหน้า* ในปี 1995 คำตอบคือ "ก็ทนอยู่กับมันไป" แต่คำตอบที่อุตสาหกรรมเลือกจริง ๆ คือ JavaScript — เริ่มจากโรยนิด ๆ หน่อย ๆ ก่อนจะกลายเป็นหิมะถล่ม

หิมะถล่มก้อนนั้นคือเรื่องของบทถัดไป แต่ก่อนจะไป เก็บรายการข้างบนนี้ไว้ใกล้ ๆ ตัว เพราะการ over-engineer ที่พบบ่อยที่สุดใน frontend ไม่ใช่การเลือก framework ผิด — แต่คือการคว้า framework *อะไรก็ได้* มาสร้างใหม่ในสิ่งที่ browser ทำให้อยู่แล้ว ฟรี ๆ แถมยังมาพร้อม accessibility ที่ดีกว่าของที่เราสร้างใหม่เสียอีก

---

**สรุป:** document architecture คือ application model ที่สมบูรณ์และมีความจริงอยู่ชุดเดียว — ส่วน bug frontend สมัยใหม่กองใหญ่ ๆ ก็คือราคาที่เราจ่ายไปเพื่อทิ้งมัน เพราะฉะนั้นจะทิ้งมันก็ต่อเมื่อได้อะไรเฉพาะเจาะจงกลับมาเท่านั้น บทถัดไป: วิธีจ่ายที่ถูกที่สุดวิธีแรก — JavaScript นิด ๆ หน่อย ๆ ที่เข้ามา enhance document ที่ทำงานได้อยู่แล้ว แทนที่จะเข้าไปแทนที่มันทั้งหมด
