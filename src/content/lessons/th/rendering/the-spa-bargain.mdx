ราวต้นทศวรรษ 2010 โลกแบบ sprinkle มีปัญหาจริง Gmail, Google Maps, Trello — โปรดักต์ที่ session ยาวเป็นชั่วโมงและ interface เปลี่ยนหลายสิบครั้งต่อนาที — โตเกินกว่า "HTML บวก event listener" การประสาน state ที่เข้มข้นผ่าน DOM ล้มเหลวเมื่อสเกลใหญ่ขึ้น สปาเกตตี้ jQuery กลายเป็นมีมตอนสัมภาษณ์งาน และการรีโหลดทั้งหน้าก็รู้สึกเหลวไหลเมื่ออยู่ในแอปที่คุณไม่เคยออกไปไหน

single-page application คือคำตอบที่เป็นเหตุเป็นผล: **โหลด application ครั้งเดียว แล้วไม่ต้อง navigate อีกเลย** server เลิกส่งหน้าแล้วเริ่มส่ง JSON; JavaScript เป็นเจ้าของ DOM อย่างถาวร; "navigation" กลายเป็นการที่แอปวาดตัวเองใหม่ router, view, state — อยู่ฝั่ง client ทั้งหมด

ข้ออ้างของบทนี้: SPA คือ *ดีลแลกเปลี่ยน* ในความหมายเก่าของคำ — การแลกเปลี่ยนที่ต่อรองกันแล้ว คุณได้บางอย่างที่มีค่าจริง ๆ คุณจ่ายด้วย feature ของ browser ที่ผู้ใช้ไม่รู้ตัวว่าเคยได้ฟรี ดีลดีหรือไม่ขึ้นกับรูปร่างของโปรดักต์ล้วน ๆ และแทบไม่มีใครตีราคามันก่อนเซ็น

## สิ่งที่คุณได้

ยุติธรรมกับ SPA หน่อย เพราะภาพล้อเลียน ("ของบวมเกินจำเป็นสำหรับบล็อก") บังว่าทำไมมันถึงชนะมาทศวรรษ:

- **ความเป็น stateful ข้าม interaction** — client state อยู่ต่อข้าม "หน้า": panel filter อยู่ที่เดิม, เพลงเล่นต่อ, คอมเมนต์ที่พิมพ์ค้างไม่หาย นี่คือ feature เด็ด และ document model ทำไม่ได้เลย
- **ความหนาแน่นของ interaction** — เมื่อ state อยู่ใน memory หนึ่ง interaction ราคาแค่ function call ไม่ใช่ round trip drag-and-drop, การทำงานร่วมกันแบบสด, UI ที่ update หกสิบครั้งต่อวินาทีกลายเป็นเรื่องธรรมชาติ
- **mental model เดียว** — UI ทั้งหมดคือโปรแกรมเดียว ภาษาเดียว component tree เดียว ใช้เหตุผลกับมันง่ายกว่า template-ตรงนี้ sprinkle-ตรงนั้นจริง ๆ — นี่ต่างหาก ไม่ใช่ performance ที่เป็นเหตุผลที่นักพัฒนาเลือกมัน
- **transition แบบไม่เปลี่ยนหน้า** — เปลือกอยู่ที่เดิม เนื้อหา morph ไม่มีแฟลชขาว ไม่ต้องสร้างตำแหน่ง scroll ใหม่ (จำไว้ เพราะคุณจ่ายค่ามันที่อื่น)

## บิลแบบแจกแจง

นี่คือสิ่งที่ browser เคยทำให้ฟรี ซึ่งตอนนี้เป็นงานของทีมวิศวกรคุณ:

**ตัว navigation เอง** — routing, ปุ่ม back, การคืนตำแหน่ง scroll, การย้าย focus ไปเนื้อหาใหม่สำหรับ screen reader, "เปิดในแท็บใหม่", URL-เป็น-state-ที่-share-ได้ — router ของคุณประมาณการแต่ละอย่าง ประมาณการ: ทุกโค้ดเบส SPA จะมี bug เรื่องการคืน scroll สักวัน และส่วนใหญ่มี bug accessibility เพราะของ browser ใช้เวลายี่สิบปีในการทำให้แกร่ง

**loading state** — MPA มี loading state เดียวและ browser วาดให้ (สปินเนอร์ในแท็บ — ผู้ใช้เข้าใจมาตั้งแต่ 1994) SPA มี loading state *n* สถานะ เมื่อ n คือจำนวนของที่ fetch — และทุกอันคือ UI ที่คุณต้องออกแบบ สร้าง และจัดลำดับ ระเบียบวินัยเรื่อง loading state มีบทของมันเองใน track async ตรงนี้แค่สังเกตว่า SPA เป็นคนสร้างปัญหานี้ขึ้นมา

**first load** — ช่วงเวลาที่เร็วที่สุดของ document model — มาถึง รับ HTML อ่าน — กลายเป็นช่วงที่ช้าที่สุดของ SPA: มาถึง ไม่ได้อะไร ดาวน์โหลดแอป boot มัน fetch ข้อมูล *แล้วค่อย* อ่าน คุณจะรู้สึกมันใน demo ข้างล่าง

**memory และเวลา** — "garbage collection" ของ document คือ navigation: ทุกอย่างถูกทำลายและสร้างใหม่สด ๆ SPA ทำงานเป็นชั่วโมง ทุก subscription ที่รั่วและ listener ที่ลืมก็สะสม โปรแกรม client ที่อยู่นานต้องการระเบียบวินัยของแอปเดสก์ท็อป ซึ่งเป็นระเบียบที่ทีมเว็บในอดีตไม่ต้องมี

**crawler และ preview** — อะไรก็ตามที่อ่าน HTML โดยไม่รัน JavaScript — search engine (ที่รันบ้างไม่รันบ้าง), ตัว unfurl ลิงก์, RSS reader — เห็นแต่เปลือก

ไม่มีอันไหนถึงตาย แต่ละอันแก้ได้ และทีม SPA ที่โตแล้วก็แก้ ประเด็นคือทุกทางแก้คือ *งานที่ทำไปตลอด* — ภาษีที่จ่ายด้วยเวลาวิศวกร — และดีลจะคุ้มก็ต่อเมื่อความเป็น stateful ที่คุณซื้อมีค่ามากกว่าภาษีนั้น

## กฎ amortization

แล้วเมื่อไรดีลถึงจะดี? ลอกวาทกรรมออกแล้วมันคือปัญหา amortization:

**ต้นทุนของ SPA จ่ายหน้า (boot) และเป็นเชิงโครงสร้าง (ภาษี) ประโยชน์ของมันเกิดต่อ interaction ดีลคุ้มเมื่อมี interaction ต่อ session มากพอที่จะ amortize ได้**

- **dashboard** ที่คนใช้สามชั่วโมง: interaction หลายพันครั้ง amortize boot ครั้งเดียว ดีลยอดเยี่ยม
- **editor ดีไซน์, email client, CRM**: ผู้ใช้อยู่ที่นั่น client state ข้าม interaction คือตัวโปรดักต์ ดีลยอดเยี่ยม
- **checkout**: ห้า interaction แล้วจากไป — แล้วคุณให้ผู้ใช้ดาวน์โหลด application มากรอกสี่ช่อง ดีลแย่
- **เว็บเนื้อหา**: หนึ่ง interaction (มาถึง อ่าน) คือทั้ง session คุณจ่ายค่า boot เพื่อส่งสิ่งที่ไฟล์ static ส่งได้เร็วกว่า ดีลที่แย่ที่สุดในเมนู และเป็นค่า default ของอุตสาหกรรมมาทั้งทศวรรษ

ข้อพ่วงสมควรมีบรรทัดของตัวเอง: **"อยู่หลัง login" เปลี่ยนสมการ** พื้นผิวที่เปิดสาธารณะ ขับด้วยการค้นหา และสำคัญเรื่องความประทับใจแรกคือที่ที่ต้นทุน SPA กระจุก (first paint, SEO, คนแปลกหน้าบนเน็ตช้า) ส่วนพื้นผิวหลัง login คือที่ที่มันระเหย — ผู้ใช้เลือกคุณแล้ว, crawler ไม่ได้รับเชิญ, และ session ยาว นี่คือเหตุผลที่ "SPA สำหรับแอป, document สำหรับทุกอย่างที่เปิดสาธารณะ" กลายเป็นค่า default ของผู้ใหญ่อย่างเงียบ ๆ ในบริษัทที่เคยเจ็บมา

## เอาไปใช้วันจันทร์ยังไง

1. **ตีราคาพื้นผิวใหม่ด้วยกฎ amortization** — interaction ต่อ session, สาธารณะหรือหลัง login, ไวต่อคนแปลกหน้าแค่ไหน สิบนาทีกับเรื่องนี้คุ้มกว่าหนึ่งไตรมาสของการมาแก้ performance ทีหลัง
2. **ถ้าอยู่ใน SPA แล้ว ตรวจดูว่า route ไหนเป็นรูปร่างแอปจริง ๆ** — SPA ส่วนใหญ่มีหน้ารูปร่าง document ไม่กี่หน้า (pricing, บล็อก, help) ที่จ่ายภาษีแอปเต็ม ๆ การแยกมันออกมักถูกและวัดผลได้ทันที
3. **จับตาเลข first-load จริง ๆ ของคุณ** — บนมือถือ Android กลาง ๆ บน 4G ไม่ใช่บนแล็ปท็อปคุณ บิลมองไม่เห็นจากเครื่อง dev ซึ่งเป็นเหตุผลที่มันค้างไม่ได้จ่ายจนกระทั่งมีการตื่นตระหนกตอน OKR รายไตรมาส

---

**สรุป:** SPA แลก feature ฟรีของ browser กับความเป็น stateful และความหนาแน่นของ interaction เป็นการแลกที่ดีจริงสำหรับ session ที่ยาว เข้มข้น และอยู่หลัง login และแย่เชิงโครงสร้างสำหรับการเข้าชมสั้น ๆ หรือรูปร่างเนื้อหา — และวิธีเดียวที่จะรู้ว่าคุณมีแบบไหนคือนับ interaction ไม่ใช่นับแต้มแฟชั่น บทถัดไป: ความพยายามของอุตสาหกรรมที่จะได้ทั้งสองฝั่งพร้อมกัน — server-side rendering, hydration และภาษีแปลก ๆ ของการ render ทุกอย่างสองรอบ
