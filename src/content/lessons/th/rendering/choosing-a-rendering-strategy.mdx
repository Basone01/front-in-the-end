สี่บทของประวัติศาสตร์-ใน-ฐานะ-ข้อโต้แย้ง บีบลงเหลือย่อหน้าเดียว: document ให้ความจริงชุดเดียวและ feature ทั้งชุดของ browser ฟรี แต่ update ได้แค่ด้วยการรีโหลด (บทหนึ่ง) JavaScript มา enhance document ที่ทำงานได้หรือแทนที่มัน และการแทนที่ควรเป็นการเลือก ไม่ใช่ default (บทสอง) SPA คือการแทนที่นั้นเต็มสเกล — ดีลที่ดีจริงสำหรับ session ที่ยาวและเข้มข้น และแย่เชิงโครงสร้างสำหรับการเข้าชมสั้นหรือรูปร่างเนื้อหา (บทสาม) SSR และ hydration เอาจุดแข็งของ document กลับมาแปะ SPA ด้วยราคาของการ render ทุกอย่างสองรอบ ราคาที่ island และ server component มีอยู่เพื่อลด (บทสี่)

ทุกเทคโนโลยีในย่อหน้านั้นคือการตอบสนองที่เป็นเหตุเป็นผลต่อกำแพงจริง ๆ แปลว่าคำถาม "อันไหนดีที่สุด?" ตั้งผิดรูป — เหมือนถามว่ารถบรรทุกดีกว่ารถเก๋งไหม คำถามที่ตั้งถูกรูปคือคำถามที่บทสุดท้ายนี้สร้างกรอบให้: **โปรดักต์คุณรูปร่างยังไง และรูปร่างนั้นทน trade-off แบบไหนได้?**

## สามคำถาม

เกือบทั้งการตัดสินใจบีบลงเหลือสามคำถามรูปร่าง-โปรดักต์ — ไม่ใช่รูปร่าง-เทคโนโลยี

**1. นี่คือเนื้อหาหรือแอป?** ผู้ใช้ทำอะไรเป็นหลัก: *อ่านสิ่งต่าง ๆ* หรือ *ใช้งานบางอย่าง*? อ่านแปลว่าหน้าที่ของหน้าคือมาถึงเร็วแล้วหลบไป — ฝั่ง document ใช้งานแปลว่ามี client state สะสมข้าม interaction หลายครั้ง — ฝั่งแอป โปรดักต์จริงส่วนใหญ่เป็นทั้งคู่ ทีละหน้า: หน้าร้านคือเนื้อหา (แคตตาล็อก) ห่อรอบแอป (checkout, ตัว config) นั่นโอเค — คำตอบเป็นต่อพื้นผิว ไม่ใช่ต่อบริษัท และ "วิธี render ต่างกันสำหรับ route ต่างกัน" คือ feature ของทุก framework จริงจังทุกวันนี้

**2. คนแปลกหน้าสำคัญไหม?** คนนิรนามที่มาจากการค้นหาและลิงก์ที่ share บริโภค HTML ตัดสินคุณในวินาทีแรก และรันโค้ดคุณบนมือถือที่คุณไม่เคยเทสต์ ถ้าพวกเขาคือธุรกิจ — commerce, เนื้อหา, การตลาด — first paint และ crawlability คือสิ่งที่ต่อรองไม่ได้ และ architecture ที่ ship เปลือกเปล่าก็ตกรอบ หลัง login ข้อจำกัดเรื่องคนแปลกหน้าหายไป และพร้อมกับมันก็หายเหตุผลส่วนใหญ่ที่จะคัดค้านการ render ฝั่ง client ล้วน ๆ

**3. session รูปร่างยังไง?** ยาวและเข้มข้น amortize ต้นทุนหน้าใด ๆ ได้ — boot bundle, hydration, ทั้งหมด (กฎบทสาม) สั้นและเป็น transaction amortize อะไรไม่ได้เลย: ทุก kilobyte จ่ายโดยผู้ใช้ที่จะจากไปในเก้าสิบวินาที

## สิ่งที่กรอบนี้จงใจละไว้ — และหนึ่งอย่างที่ไม่ควรละ

ข้อแม้ซื่อสัตย์สามข้อก่อนจะเคลือบต้นไม้ตัดสินใจ

**ทีมของคุณคือ input จริง** ทีมผู้เชี่ยวชาญ React ที่ ship เว็บ Next.js หนักไปนิดจะชนะทีมที่เพิ่งเรียน islands architecture ใต้เส้นตาย ของที่น่าเบื่อแต่รู้จักชนะเสมอตอนเสมอ — *ตอนเสมอ* สิ่งที่มันไม่ชนะคือความไม่เข้ากันเชิงโครงสร้าง: ไม่มีความคุ้นเคยของทีมเท่าไรที่ทำให้ boot 400 KB เหมาะกับ checkout รู้ว่าคุณกำลังดูการตัดสินใจแบบไหน

**ต้นทุนการ migrate เป็นเรื่องจริง** กรอบข้างบนตีราคาการตัดสินใจ greenfield การ re-architect โปรดักต์ที่ทำงานอยู่ตีราคาเป็นไตรมาส และ "บีบทีละ route — แยกหน้ารูปร่าง document ที่สุดออกมาก่อน" (คำแนะนำวันจันทร์ของบทสาม) ชนะการเขียนใหม่เกือบทุกครั้ง

**ค่า default เหนียวแน่นโดยตั้งใจ** framework ของเจ้าต่าง ๆ ตั้ง default เป็นโหมดทรงพลังที่สุด งานสัมมนาโชว์ของใหม่ และ "เราใช้ X กับทุกอย่าง" ถูกกว่าเชิงองค์กรเทียบกับการตัดสินใจต่อพื้นผิว การเลือกวิธีที่ทรงพลัง *น้อยที่สุด* ที่เหมาะจึงเป็นความกล้าหาญเชิงองค์กรเล็ก ๆ เผื่องบสำหรับการเถียง ไม่ใช่แค่การสร้าง

## นิสัยที่อยู่นานกว่าคำตอบ

track นี้ใช้การ render สอนอะไรที่ทั่วไปกว่านั้น และน่าเสียดายถ้าจะปล่อยให้มันเป็นนัยอยู่เฉย ๆ

ทุกชั้นของเทคโนโลยี frontend มีอยู่เพราะชั้นใต้มันชนกำแพงเฉพาะอย่าง แปลว่าทุกชั้นมี *จุดประสงค์* — ปัญหาที่มันแก้ — และ *ราคา* — ของฟรีที่มันยกให้ไป ทักษะวิศวกรรมไม่ใช่การรู้จักชั้นต่าง ๆ ตลาดงานเต็มไปด้วยคนที่รู้จักชั้น ทักษะคือ **การจับคู่ชั้นกับปัญหาและปฏิเสธที่จะจ่ายค่าพลังที่คุณจะไม่ใช้** — และพูดออกมาได้ดัง ๆ ในห้องประชุมว่าราคาของตัวเลือกแวววาวนั้นคือเท่าไร

นิสัยนั้นโอนถ่ายได้ เมื่อคุณเจอ abstraction ตัวถัดไป — state manager, library ดึงข้อมูล, framework-ที่-AI-สร้าง-ทุกอย่าง — คำถามอยู่ในมือคุณแล้ว: *นี่พาฉันข้ามกำแพงอะไร? ฉันเคยได้อะไรฟรีที่ตอนนี้กำลังซื้อกลับ?* ถ้าไม่มีใครในห้องตอบได้ นั่นแหละคือคำตอบ

track ถัดไปใช้นิสัยเดียวกันภายในแอปที่กำลังรัน: state ควรอยู่ที่ไหนและ "source of truth" ให้อะไร และ network ทำอะไรกับ UI เมื่อ request จบไม่เรียงลำดับ วิธีเดียวกัน — รู้สึกถึงกำแพง แล้วค่อยตีราคา workaround

**สรุป:** เนื้อหาหรือแอป, คนแปลกหน้าหรือขาประจำ, สั้นหรือยาว — ตอบต่อพื้นผิวแล้ววิธี render ก็เลือกตัวเองเป็นส่วนใหญ่: static สำหรับของที่อ่าน, sprinkle และ island สำหรับของที่ interactive เบา ๆ, SPA สำหรับของที่คนอยู่กับมัน, SSR + hydration เฉพาะที่คนแปลกหน้ากับความเป็น stateful ชนกัน ใช้วิธีที่ทรงพลังน้อยที่สุดที่เหมาะ และเมื่อคว้าพลังเพิ่ม ให้พูดได้ว่ามันราคาเท่าไร ประโยคนั้นคือทั้ง track นี้
