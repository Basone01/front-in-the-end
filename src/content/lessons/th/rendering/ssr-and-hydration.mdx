ดีลของ SPA ทิ้งเหยื่อไว้รายหนึ่ง: คนแปลกหน้า คนที่มาจากผลการค้นหา บนมือถือ บน wifi รถไฟฟ้า ที่ได้รับ `<div>` เปล่า ๆ กับการบ้าน 480 KB สำหรับเครื่องมือหลัง login ไม่มีใครสน แล้ว SPA ก็กลายเป็นค่า default ของ *ทุกอย่าง* — หน้าร้าน เว็บข่าว landing page — แล้วเหยื่อก็เริ่มโผล่ในแดชบอร์ดรายได้

คำตอบของอุตสาหกรรม: **render แอปที่ server ด้วย** รัน component เดียวกันใน Node, ผลิต HTML จริง, ส่งไป คนแปลกหน้าเห็นเนื้อหาทันที — เร็วแบบ document — *แล้ว* JavaScript ค่อยมาถึงและเข้า takeover เนียนแบบ SPA ได้ทั้งสองโลก ลูกตุ้มที่แกว่งจาก document ที่ render ฝั่ง server ไปสู่แอปที่ render ฝั่ง client ก็แกว่งกลับมาทางกลาง

ข้ออ้างของบทนี้: SSR ทำงานได้จริง — เป็นทางเลือกที่ถูกสำหรับโปรดักต์กลุ่มใหญ่ — แต่ **hydration กลไกที่ทำให้มันทำงาน ไม่ใช่ feature มันคือบิล** การเข้าใจมันในฐานะบิลแทนที่จะเป็นเวทมนตร์คือสิ่งที่ทำให้คุณตัดสินใจได้ว่าเมื่อไรมันคุ้มที่จะจ่าย

## render สองรอบ

กลไก ลอกเหลือแต่โครง:

1. **Server:** รัน component พร้อมข้อมูล ผลิต HTML ส่งไป
2. **Browser:** วาด HTML นั้นทันที ผู้ใช้เห็นหน้าที่สมบูรณ์
3. **Browser:** ดาวน์โหลด bundle JavaScript — ซึ่งมี *component ชุดเดียวกัน*
4. **Hydration:** รัน component *อีกครั้ง* ใน browser ได้ผลลัพธ์เสมือนที่ควรตรงกับ HTML ที่อยู่บนจอแล้ว อย่าวาดใหม่ — แต่เดิน DOM ที่มีอยู่แล้วผูก event handler เข้าไปกับมัน รับมันมาเป็นของแอป
5. จากนี้ไป: SPA ปกติ

อ่านข้อ 4 อีกครั้ง เพราะมันคือกลไกที่แปลกที่สุดใน frontend กระแสหลัก: client คำนวณผลลัพธ์ที่ server คำนวณไปแล้วใหม่ตั้งแต่ต้น — ไม่ใช่เพื่อแสดง (มันแสดงอยู่แล้ว) แต่เพื่อ *สร้าง state ใน memory ที่ SPA ควรจะมี* ถ้ามัน render หน้านั้นเอง งานถูกทำซ้ำโดยตั้งใจ "hydration" ฟังดูเหมือนพรมน้ำใส่อะไรบางอย่าง แต่ความจริงใกล้กับการสร้างเครื่องยนต์เรือใหม่ขณะที่ผู้โดยสารอยู่บนเรือแล้ว

ผลที่ตามมาสองข้อหลุดออกมาตรง ๆ จากกลไกนี้

## ผลที่ตามมาข้อหนึ่ง: uncanny valley

ระหว่างข้อ 2 (เนื้อหามองเห็น) กับข้อ 4 (handler ผูกแล้ว) มีช่วงที่ **หน้ามองดูเสร็จแล้วและกำลังโกหกเรื่องนั้น** ปุ่ม render ออกมา มองดู enabled และไม่ทำอะไรเลย ยิ่ง server เร็วและเน็ต/CPU ช้า ช่วงนี้ยิ่งกว้าง

## ผลที่ตามมาข้อสอง: คุณ ship ทุกอย่างสองรอบ

โค้ด component ต้องมีอยู่ที่ server (เพื่อ render HTML) *และ* ใน bundle (เพื่อ hydrate) SSR ไม่ได้ลด JavaScript ของ SPA — มันเพิ่มการ render ที่ server เข้าไปข้างหน้ามัน first *paint* ดีขึ้นมหาศาล แต่ time to *interactive* ดีขึ้นนิดเดียว หรือไม่ดีขึ้นเลย ถ้าปัญหาคือ "ผู้ใช้รอนานเกินกว่าจะเห็นเนื้อหา" SSR แก้ได้ ถ้าปัญหาคือ "เรา ship JavaScript เยอะเกินไป" SSR คือวิธีรู้สึกดีขึ้นกับมัน

มีโหมดพังใหม่แถมมาด้วย: **hydration mismatch** ถ้า client render ไม่ตรงกับ HTML ของ server — timestamp ที่จัด format คนละ timezone, `Math.random()`, อะไรก็ตามที่อ่าน `window` ตอน render — framework ต้องประนีประนอมความจริงสองเวอร์ชัน ผลตั้งแต่ warning ใน console ไปจนถึง UI ที่เพี้ยนแบบแนบเนียน การ render สองรอบแปลว่าต้อง *ตรงกัน* สองรอบ และ component ต้องเขียนให้ render เหมือนกันในสองสภาพแวดล้อม ระเบียบนั้นคือภาษีบน component ทุกตัวที่คุณเขียน ไม่ว่ามันจะเคยทำตัวมีปัญหาหรือไม่

**เช็กคำศัพท์** SSR (render ต่อ request ที่ server), SSG (render ตอน build — สิ่งที่เว็บนี้ทำ), และ ISR (render ตามต้องการแล้ว cache) ต่างกันที่ *เมื่อไร* ที่ HTML ถูกผลิต hydration เป็นเรื่องตั้งฉาก: มันคือสิ่งที่เกิด *หลัง* HTML มาถึง และบิลของมันเท่ากันทั้งสามแบบ หน้า static ที่ hydrate แอปเต็ม ๆ ก็จ่ายภาษี hydration เต็มจาก CDN ฟรี

## ลดบิล: islands และ server components

ถ้าต้นทุน hydration เป็นสัดส่วนกับว่าหน้า hydrate มากแค่ไหน ท่าที่ชัดเจนคือ: hydrate ให้น้อยลง นวัตกรรม framework ห้าปีที่ผ่านมาส่วนใหญ่คือประโยคนี้

**Islands** (Astro, Fresh, Eleventy + component แยก): หน้าคือ HTML static ยกเว้นโซน interactive ที่ประกาศไว้ แต่ละโซน hydrate แยกกัน carousel เป็น island ส่วนบทความรอบ ๆ ไม่ ship JavaScript เลย เว็บนี้ทำแบบนั้นเป๊ะ — demo ข้างบนเป็น island ส่วนเนื้อหาที่คุณอ่านอยู่ไม่เคย hydrate mental model คือเส้นความเป็นเจ้าของของบทสอง วาดต่อ component: document เป็นเจ้าของหน้า, JavaScript เป็นเจ้าของ island ของมัน

**React Server Components** วาดเส้นเดียวกันใน component tree: server component รัน *เฉพาะ* ที่ server — โค้ดมันไม่เข้า bundle, แตะ database ตรง ๆ ได้, re-render โดยถาม server — ส่วน client component hydrate ตามปกติ RSC คือ React ยอมรับสมมติฐานของทั้ง track นี้: หน้าทั่วไปส่วนใหญ่ไม่เคยจำเป็นต้องเป็น client application

พูดอีกอย่าง ลูกตุ้มไม่ได้แกว่งไปมาอีกแล้ว มันกำลังลู่เข้า — สู่ hydration แบบบางส่วน ตั้งใจ และต่อ component สงคราม "document vs apps" จบลงที่การต่อรองเส้นแบ่ง และวิศวกรที่วาดเส้นนั้นได้ดีคือคนที่ track นี้พยายามผลิต

## เมื่อไรควรจ่ายบิล SSR

กฎการตัดสินใจสั้นกว่าชื่อเสียงของมัน SSR + hydration คุ้มความซับซ้อนเมื่อ **ทั้งสอง** ข้อนี้จริง:

1. **คนแปลกหน้าสำคัญ** — พื้นผิวสาธารณะ ขับด้วยการค้นหา share ลิงก์ สำคัญเรื่อง first-paint
2. **หน้านี้ต้องการ interactivity ระดับแอปจริง ๆ หลัง paint** — ไม่ใช่เมนู nav กับ form (sprinkle และ island ครอบคลุมด้วยราคาเศษเสี้ยว) แต่เป็น client statefulness จริง ๆ แบบบทสาม

หน้าสินค้า e-commerce คือกรณีจริงทั้งสองข้อแบบฉบับ: SEO คือธุรกิจ และหน้าหนาแน่นด้วย UI ที่ stateful dashboard หลัง login ตกข้อ 1 — CSR ล้วนง่ายกว่าและไม่มีใครเดือดร้อน บล็อกตกข้อ 2 — HTML static กับ island สักอันก็พอ ข้าม runtime framework ไปเลย

และถ้าคุณจ่ายบิลจริง **ออกแบบเผื่อ uncanny valley**: render `<form>` และ `<a>` จริง ๆ ที่ server เพื่อให้หน้าก่อน hydrate ลดระดับลงเป็นพฤติกรรม document ที่ทำงานได้ (บทสองอีกครั้ง) แทนที่จะเป็น pixel ที่ตายแล้ว uncanny valley อยู่รอดได้ การแตะที่ตายเงียบ ๆ ไม่รอด

---

**สรุป:** SSR เอาพลังพิเศษของ document — เนื้อหาใน response แรก — กลับมาแปะ SPA และ hydration คือราคาที่งุ่มง่ามและ render สองรอบ จ่ายมันเมื่อคนแปลกหน้าและความเป็น stateful มาบรรจบกัน ลดมันด้วย island หรือ server component เมื่อมันไม่บรรจบ และอย่าเข้าใจผิดว่า "painted" คือ "เสร็จ" — ผู้ใช้คุณไม่เข้าใจผิดแน่ ๆ บทสุดท้ายของ track: พับทั้งสี่บทเข้าเป็นการตัดสินใจเดียวที่ ship จริง — เลือกวิธี render อย่างตั้งใจ
